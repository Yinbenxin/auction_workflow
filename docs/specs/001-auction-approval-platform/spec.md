# 001 - 竞拍工作平台 Spec

## 功能概述

构建一个覆盖竞拍全流程的工作平台，支持以下 10 个阶段的流程管理和留痕：竞拍信息收集、历史数据分析、策略方案制定、策略评审确认、任务配置、执行前复核、正式竞拍执行、实时监控、异常修改留痕、结果复盘与策略迭代。平台以"负责人确认留痕"为核心机制，确认是流程推进的门控，每次确认均记录操作人、时间和内容，实现全程可追溯、可复盘。

---

## 背景（澄清结果）

- **功能边界：** 全部 10 个阶段一次性实现：01 竞拍信息收集、02 历史数据分析、03 策略方案制定、04 策略评审确认、05 任务配置、06 执行前复核、07 正式竞拍执行、08 实时监控、09 异常修改留痕、10 结果复盘与策略迭代。
- **权限角色：** 业务负责人、策略负责人、审核人、交易员、复核人、数据人员、监控人员、复盘负责人，共八类角色。其中"审核人"独立于策略负责人，负责对策略版本进行审核把关，策略提交人与审核人必须为不同账号。
- **确认机制：** 采用分级确认留痕机制，根据策略风险等级决定确认人集合：普通策略需策略负责人提交 + 审核人确认；涉及红线字段变更（价格/数量/时点/触发条件/兜底方案/适用场景）需额外加业务负责人确认。系统记录每次确认的操作人、时间、内容，实现全程留痕。确认是流程推进的门控，各阶段阻断规则见业务规则章节。
- **并发与幂等：** 乐观锁或状态前置校验防止并发冲突，保证操作幂等性。
- **异常处理：** 红线字段（价格/数量/时点/触发条件/兜底方案/适用场景）变更后必须重新走确认流程；作废版本保留不删除。

## 用户故事

### 阶段 01：竞拍信息收集

**US-01 录入竞拍基础信息**
作为数据人员，我希望能录入竞拍公告、规则、起止时间、投放量、需求量、限价规则、申报单位等基础信息，以便为后续策略制定提供准确的信息基础。

**US-02 确认竞拍基础信息**
作为业务负责人，我希望能查看并确认竞拍基础信息表，系统记录确认人和时间，以便确保信息完整准确后再进入下一阶段。

**US-03 查看竞拍信息收集状态**
作为任意角色，我希望能查看当前竞拍项目的信息收集完成状态，以便了解是否可以进入历史数据分析阶段。

### 阶段 02：历史数据分析

**US-04 录入历史竞拍分析数据**
作为数据人员，我希望能录入历史成交价格、抢量速度、时间点变化、异常情况等分析维度数据，以便为策略制定提供数据支撑。

**US-05 确认历史数据分析结果**
作为策略负责人，我希望能查看并确认历史竞拍分析表，系统记录确认人和时间，以便基于数据而非经验制定策略。

### 阶段 03：策略方案制定

**US-06 创建策略版本**
作为策略负责人，我希望能为指定竞拍项目创建新的策略版本，填写价格、数量、时点、触发条件、兜底方案、适用场景等核心字段，以便形成结构化、可执行的策略方案。

**US-07 查看版本历史**
作为任意角色，我希望能查看某竞拍项目下所有策略版本的历史列表及每个版本的完整确认记录，以便了解策略演变过程和责任链。

**US-08 作废策略版本**
作为策略负责人，我希望能将不再使用的策略版本标记为作废，同时保留历史记录，以便在出现争议时仍可追溯原始内容。

### 阶段 04：策略评审确认

**US-09 提交策略版本待确认**
作为策略负责人，我希望将策略版本提交给对应负责人确认，系统自动判断需要哪些角色确认（普通策略/涉及红线字段），以便确保策略经过适当的确认后再执行。

**US-10 负责人确认策略版本**
作为业务负责人或策略负责人（审核角色），我希望能查看待确认的策略详情，点击"确认"或"驳回"（附驳回原因），系统记录操作人和时间，以便对策略变更进行有效把关。

**US-11 标记最终版本**
作为策略负责人，我希望在确认通过后将某一版本标记为"最终版本"，以便后续执行前复核和复盘报告能明确关联到唯一的执行依据。

### 阶段 05：任务配置

**US-12 录入任务配置清单**
作为交易员，我希望能按最终确认策略录入完整的任务配置清单，包括：任务编号、任务时间、申报价格、申报数量、触发条件、任务顺序、任务启停状态、垫子策略、补量策略、兜底任务，并上传配置截图或导出文件作为证据材料，以便留存完整配置记录供复核使用。

**US-13 确认任务配置完成**
作为交易员，我希望在配置完成后提交确认，系统记录配置人和时间，以便进入执行前复核阶段。

### 阶段 06：执行前复核

**US-14 发起执行前复核**
作为交易员，我希望在执行竞拍任务前，能基于最终确认版本发起执行前复核流程，以便确保系统配置与策略完全一致后再执行。

**US-15 逐项确认复核清单**
作为复核人，我希望能逐项勾选 12 项复核清单，并在全部通过后提交复核结论，以便留存完整的复核痕迹。

**US-16 复核通过后解锁执行**
作为系统，当复核人提交通过结论后，应解锁交易员的"标记可执行"操作；若复核未通过，应阻止该操作，以便从系统层面保障执行前复核的有效性。

### 阶段 07：正式竞拍执行

**US-17 记录竞拍执行日志**
作为交易员，我希望能在竞拍执行过程中记录任务触发情况、申报价格、申报数量、系统状态等关键信息，以便形成完整的执行日志。

**US-18 标记竞拍执行完成**
作为交易员，我希望在竞拍结束后标记执行完成，系统记录完成时间，以便进入实时监控结果汇总和复盘阶段。

### 阶段 08：实时监控

**US-19 记录监控数据**
作为监控人员，我希望能记录竞拍过程中的价格变化、剩余量、成交速度、数据回传状态、系统响应状态等监控指标，以便为复盘提供完整的过程数据。

**US-20 记录异常事件**
作为监控人员，我希望能在发现异常时快速记录异常类型、发现时间、处理动作，以便留存异常处理的完整证据链。

### 阶段 09：异常修改审批与留痕

**US-21 提交临场修改申请**
作为交易员或监控人员，我希望在需要修改核心参数时，必须填写修改原因、修改前内容、修改后内容、影响范围，并说明是否属于预授权应急方案范围，以便提交给策略负责人审批，确保修改经过授权后再执行。

**US-22 策略负责人审批临场修改**
作为策略负责人，我希望能查看临场修改申请详情，执行审批通过或驳回（附驳回原因），系统记录审批人和时间，以便对临场修改进行有效把关。

**US-23 复核人核对临场修改参数**
作为复核人，我希望在审批通过后对修改涉及的价格、数量、时点、触发条件进行核对确认，系统记录复核人和时间，以便确保修改内容准确无误后再执行。

**US-24 执行并留痕临场修改**
作为交易员，我希望在审批和复核均通过后执行修改，并在系统中标记已执行，以便形成完整的"申请→审批→复核→执行"证据链。

**US-25 紧急应急执行与事后补说明**
作为交易员，当倒计时阶段来不及完整审批时，我希望能在预授权应急方案范围内执行紧急修改，系统标记为应急执行，并在竞拍结束后强制要求补充审批说明，以便确保应急执行有据可查。

### 阶段 10：结果复盘与策略迭代

**US-23 创建复盘报告**
作为复盘负责人，我希望在竞拍结束后创建复盘报告，并关联本次竞拍的最终确认版本，以便将执行结果与策略进行对比分析。

**US-24 填写复盘报告必填项**
作为复盘负责人，我希望系统强制要求填写复盘报告的所有必填项（基础信息、策略版本、策略摘要、执行情况、成交结果、偏差分析、异常记录、确认记录、问题原因、改进措施、策略沉淀），以便确保复盘报告的完整性。

**US-26 创建整改事项**
作为复盘负责人，我希望能在复盘报告中为每个发现的问题创建独立的整改事项，填写问题描述、根因类型、责任人、整改措施、截止时间，以便明确每个问题的整改责任和完成时间。

**US-27 跟踪整改完成状态**
作为业务负责人或复盘负责人，我希望能查看所有整改事项的完成状态（待整改/整改中/已完成），并在完成时上传证据材料，以便实现整改闭环管理。

**US-25 提交并归档复盘报告**
作为复盘负责人，我希望在复盘报告填写完整、整改事项已创建后提交归档，以便形成可供后续竞拍参考的策略沉淀记录。

### 竞拍流程阶段状态机

阶段推进规则：各阶段有明确的阻断类型（硬阻断=前置未完成则拒绝推进；软阻断=提示但允许继续）。

| 阶段推进 | 阻断类型 | 门控条件 |
|---------|---------|---------|
| 阶段01 → 阶段02 | 硬阻断 | 基础信息已由业务负责人确认 |
| 阶段02 → 阶段03 | 软阻断 | 历史分析未确认时提示，允许继续但记录跳过原因 |
| 阶段03 → 阶段04 | 硬阻断 | 策略必填字段完整，版本号不重复 |
| 阶段04 → 阶段05 | 硬阻断 | 存在最终版本（FINAL） |
| 阶段05 → 阶段06 | 硬阻断 | 任务配置已由交易员确认 |
| 阶段06 → 阶段07 | 硬阻断 | 执行前复核已通过（12项全部勾选） |
| 阶段07 → 阶段10 | 硬阻断 | 竞拍执行已标记完成 |
| 阶段10 归档 | 硬阻断 | 关联最终版本、必填项完整、未确认应急修改已补说明 |

阶段08（实时监控）和阶段09（异常修改审批）在阶段07执行中并行发生，不是执行完成后的独立阶段：

```
阶段06 执行前复核（待复核→已通过）
  └─ 复核通过 ──► 阶段07 正式执行（待执行→执行中→已完成）
                       ├─ 执行中并行 ──► 阶段08 实时监控（持续记录）
                       ├─ 执行中按需 ──► 阶段09 异常修改审批（申请→审批→复核→执行→留痕）
                       └─ 执行完成 ──► 阶段10 结果复盘（草稿→已归档）
```

### 策略版本状态机

```
草稿(DRAFT)
  └─ 提交确认 ──► 待确认(PENDING)
                    ├─ 驳回 ──► 草稿(DRAFT)（可修改后重新提交）
                    └─ 确认通过 ──► 已确认(CONFIRMED)
                                      ├─ 标记最终版本 ──► 最终版本(FINAL)
                                      │                     └─ 作废 ──► 已作废(VOIDED)
                                      └─ 作废 ──► 已作废(VOIDED)
```

**状态转换约束：**
- 已确认（CONFIRMED）或最终版本（FINAL）状态下，不允许直接修改内容，只能创建新版本。
- 已作废（VOIDED）状态为终态，不可恢复，不可物理删除。
- 同一项目同一时间只能有一个最终版本（FINAL）处于有效状态。

### 红线字段

红线字段分两类，任意字段变更均须进入对应管控流程：

#### 策略红线字段（变更后必须重新走策略确认流程）

1. 申报价格
2. 申报数量
3. 申报时点
4. 触发条件
5. 兜底方案
6. 策略适用场景

涉及策略红线字段变更时，需额外加业务负责人确认（三人确认：策略负责人提交 + 审核人确认 + 业务负责人确认）。

#### 执行配置红线字段（变更后必须进入临场修改审批流程）

1. 任务顺序
2. 任务启停状态
3. 垫子策略
4. 补量策略
5. 兜底任务
6. 任务时间
7. 任务价格
8. 任务数量
9. 触发条件

### 临场修改状态机

```
草稿(DRAFT)
  └─ 提交申请 ──► 待审批(PENDING_APPROVAL)
                    ├─ 驳回 ──► 已驳回(REJECTED)（终态）
                    └─ 审批通过 ──► 待复核(PENDING_REVIEW)
                                      ├─ 复核不通过 ──► 已驳回(REJECTED)（终态）
                                      └─ 复核通过 ──► 已审批(APPROVED)
                                                          └─ 执行修改 ──► 已执行(EXECUTED)
                                                                            └─ （应急场景）补充说明 ──► 已补说明(POST_EXPLAINED)
```

**紧急应急规则：**
- 策略方案中可预先定义 `pre_authorized_actions`（预授权应急方案），明确每种异常对应的兜底动作。
- 倒计时阶段来不及完整审批时，交易员只能在预授权范围内执行紧急修改，系统标记 `is_pre_authorized=true`。
- 不在预授权范围内的紧急修改，系统标记 `is_pre_authorized=false`，复盘归档前必须强制标记为流程偏差并填写原因。
- 所有应急执行，复盘报告提交前必须补充审批说明，否则系统拒绝归档。

### 双人复核规则

- 执行前复核必须由两名不同人员完成：配置人（交易员）和复核人。
- 系统在提交复核时校验配置人账号与复核人账号，若为同一账号则拒绝提交。
- 复核人必须在系统中逐项勾选以下 12 项复核清单，不允许仅口头确认：
  1. 策略版本是否为最终确认版本
  2. 任务数量是否完整
  3. 任务时间是否正确
  4. 申报价格是否正确
  5. 申报数量是否正确
  6. 触发条件是否正确
  7. 垫子策略是否配置正确
  8. 兜底策略是否配置正确
  9. 系统/插件是否正常
  10. 数据回传是否正常
  11. 交易员是否确认执行口径
  12. 复核人是否确认通过
- 12 项全部勾选通过后，复核人方可提交复核结论。

### 策略版本命名规则

版本命名格式：`项目名称_竞拍日期_策略版本`，例如：`重庆忠润_20260428_v1.0`

| 版本类型 | 命名示例 | 说明 |
|---------|---------|------|
| 初始版本 | v1.0 | 项目首个策略版本 |
| 小幅调整 | v1.1 | 非红线字段的小幅修改 |
| 重大调整 | v2.0 | 红线字段变更或结构性调整 |
| 作废版本 | v1.0-作废 | 已作废的历史版本 |
| 最终版本 | v2.0-最终版 | 当前执行依据版本 |

同一项目同一竞拍日期下，版本号必须唯一，系统需在创建时校验并拒绝重复版本号。

### 核心验收标准（Given/When/Then）

**策略版本管理**
- Given 策略版本处于 CONFIRMED 或 FINAL 状态，When 用户尝试直接修改内容，Then 系统拒绝并提示"请创建新版本进行修改"。
- Given 新版本涉及策略红线字段变更，When 提交确认，Then 系统要求审核人 + 业务负责人双重确认，缺一不可。
- Given 同一项目同一日期已存在版本号 v1.0，When 创建相同版本号，Then 系统拒绝并提示版本号重复。

**执行前复核**
- Given 配置人账号与复核人账号相同，When 提交复核，Then 系统拒绝并提示"配置人与复核人不能为同一账号"。
- Given 12 项复核清单中有任意一项未勾选通过，When 提交复核结论，Then 系统拒绝提交。
- Given 执行前复核未通过，When 交易员尝试标记任务为"可执行"，Then 系统拒绝并提示"请完成执行前复核"。

**临场修改审批**
- Given 交易员提交临场修改申请，When 修改原因或影响范围为空，Then 系统拒绝提交。
- Given 临场修改未经审批通过，When 交易员尝试执行修改，Then 系统拒绝并提示"请等待审批通过后再执行"。
- Given 应急执行标记为非预授权范围，When 提交复盘归档，Then 系统强制要求标记流程偏差并填写原因。

**复盘归档**
- Given 竞拍项目下不存在最终版本（FINAL），When 提交复盘报告，Then 系统拒绝并提示"请先确认最终策略版本"。
- Given 存在未补说明的应急执行记录，When 提交复盘归档，Then 系统拒绝并提示"请补充应急执行审批说明"。
- Given 复盘报告任意必填项为空，When 提交归档，Then 系统拒绝并列出未填写的字段名。

## Edge Cases

**EC-1：策略版本状态非法流转**
已确认（CONFIRMED）或最终版本（FINAL）状态的策略版本，不允许直接修改内容。用户尝试修改时，系统应拒绝操作并提示"请创建新版本进行修改"。已作废（VOIDED）版本不允许物理删除，仅保留历史记录。

**EC-2：红线字段变更未重新确认**
申报价格、申报数量、申报时点、触发条件、兜底方案、策略适用场景任意一项发生变更时，系统必须强制触发重新确认流程。若检测到红线字段变更但未走确认流程，系统应拒绝提交并提示需重新确认。

**EC-3：双人复核同一人**
执行前复核提交时，若配置人账号（交易员）与复核人账号相同，系统必须拒绝提交，并返回错误提示"配置人与复核人不能为同一账号，请由不同人员完成复核"。

**EC-4：未完成执行前复核强行进入执行**
交易员尝试将任务标记为"可执行"时，系统必须校验执行前复核状态。若复核流程未完成或复核结论为未通过，系统应拒绝操作并提示"请完成执行前复核后再标记为可执行"。

**EC-5：临场紧急执行事后补确认遗漏**
当竞拍记录中存在临场修改记录且未经确认时，复盘报告提交前系统必须校验是否已填写应急执行的说明。若未填写，系统应拒绝提交复盘报告并提示"存在未确认的临场修改记录，请补充说明后再提交"。

**EC-6：并发操作冲突**
多人同时对同一对象（策略版本、复核记录等）执行操作时，系统通过乐观锁或状态前置校验检测冲突。后提交的操作若发现对象状态已被更新，应返回冲突提示并要求用户刷新后重新操作，不允许静默覆盖。

**EC-7：已执行版本发现错误**
已处于最终版本（FINAL）状态且已关联执行记录的版本，发现错误后只能通过作废当前版本并创建新版本的方式处理。系统不允许回滚已执行状态，不允许修改已执行版本的内容。

**EC-8：复盘报告关联版本缺失**
提交复盘报告时，系统必须校验是否已关联最终确认版本（FINAL）。若当前竞拍项目下不存在最终版本，系统应拒绝提交复盘报告并提示"请先确认最终策略版本后再提交复盘报告"。

**EC-9：策略版本号重复**
同一竞拍项目、同一竞拍日期下，版本号必须唯一。创建新版本时，系统应校验版本号是否已存在，若重复则拒绝创建并提示"该版本号已存在，请使用不同的版本号"。

**EC-10：历史数据分析未完成直接制定策略**
阶段 02（历史数据分析）未完成确认时，用户尝试进入阶段 03（策略方案制定）时，系统应给出提示"历史数据分析尚未确认，建议完成后再制定策略"，但不强制阻断，允许用户继续操作。

**EC-11：实时监控数据回传异常**
监控人员记录数据回传异常时，系统应在监控记录中标记异常状态，并在执行日志中关联异常事件，以便复盘时能追溯异常发生的时间点和处理动作。

**EC-12：异常修改未留痕直接执行**
执行中修改核心参数（价格、数量、时间点、触发条件）时，系统必须强制要求填写修改原因和影响范围，不允许空原因提交。修改记录必须关联操作人账号和时间戳，不可事后修改。

## 不在范围内

以下内容明确不在第一版交付范围内：

1. **消息通知系统** — 邮件、即时通讯工具（如飞书/钉钉）的确认通知集成，不在本期范围（系统内待办列表除外）。
2. **数据导出与报表** — 复盘报告的 PDF/Excel 导出、管理层汇总报表，不在本期范围。
3. **策略模板库** — 可复用的策略模板管理、历史策略一键复制，不在本期范围。
4. **移动端适配** — 移动端专项适配和原生 App，不在本期范围。
5. **第三方系统集成** — 与交易系统、风控系统的自动化数据对接，不在本期范围。
6. **多租户/多团队隔离** — 跨团队的数据隔离和权限管理，不在本期范围。
7. **自动化竞拍执行** — 系统自动触发竞拍任务，不在本期范围（本期只做人工操作留痕）。
8. **历史数据可视化** — 历史竞拍数据的图表分析、趋势展示，不在本期范围。
9. **策略效果评估模型** — 基于历史数据的策略效果自动评分，不在本期范围。
10. **操作审计日志** — 系统级别的所有操作审计日志（超出业务留痕范围的技术审计），不在本期范围。
